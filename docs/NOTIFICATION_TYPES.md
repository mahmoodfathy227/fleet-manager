# Notification Types Reference

Every notification is stored as a row in the `notifications` table. The `details` JSONB column carries all type-specific payload. This document is the single source of truth for what each type means, who produces it, who receives it, and what shape the `details` object has.

---

## `notifications` table — core columns

| Column | Type | Notes |
|--------|------|-------|
| `id` | `integer` | Auto-increment PK |
| `notification_type` | `varchar` | One of the types below |
| `recipient_user_id` | `uuid` (nullable) | Supabase `auth.users.id`. NULL = admin-only (dashboard only) |
| `status` | `varchar` | `pending` / `unread` / `read` |
| `details` | `jsonb` | Type-specific payload — see below |
| `created_at` | `timestamp` | Set automatically |

The 6 legacy columns (`certificate_type`, `certificate_name`, `expiry_date`, `days_until_expiry`, `entity_type`, `entity_id`) are all **nullable** and only used by the two oldest notification types (`certificate_expiry`, `vehicle_breakdown`, `driver_tardiness`). Do not populate them in new notification types — use `details` only.

---

## Push delivery rules

- Rows with a **non-null `recipient_user_id`** are automatically forwarded to Firebase FCM by the `push-notification-relay` Supabase Edge Function (triggered by a DB webhook on INSERT).
- Rows with `recipient_user_id = NULL` are **admin-only** — visible on the dashboard only.
- The relay calls `notifyInternal` Firebase Function, which writes a Firestore inbox entry and sends FCM to all registered devices for that user.
- When adding a new type that has a non-null `recipient_user_id`, you **must** add a corresponding `case` to `buildMessage()` in `supabase/functions/push-notification-relay/index.ts`, then redeploy the function.

## Type catalogue

---

### `certificate_expiry`

**What it means:** A compliance certificate (vehicle, driver, or PA) has expired or is approaching expiry. Generated by the legacy `create_certificate_notifications()` RPC.

**Produced by:** `create_certificate_notifications()` RPC  
**recipient_user_id:** NULL (admin-only)  
**Visible to:** Admins (RBAC)  
**Push sent:** No

**`details` shape:**
```json
{}
```
> Payload lives entirely in the legacy columns (`certificate_type`, `certificate_name`, `expiry_date`, `days_until_expiry`, `entity_type`, `entity_id`). No `details` content — legacy pattern, do not copy.

---

### `vehicle_breakdown`

**What it means:** A driver has reported a vehicle breakdown during an active route session.

**Produced by:** `report_vehicle_breakdown()` RPC (`supabase/migrations/072_add_vehicle_breakdown_reporting.sql`)  
**recipient_user_id:** NULL (admin-only)  
**Visible to:** Admins (RBAC)  
**Push sent:** No

**`details` shape:**
```json
{
  "breakdown_id": 42,
  "route_session_id": 7,
  "route_id": 3,
  "description": "Flat tyre on rear axle",
  "location": "High Street junction",
  "reported_at": "2026-04-07T08:15:00Z"
}
```

> **Note:** Legacy columns (`certificate_type = 'breakdown'`, `certificate_name = 'Vehicle Breakdown Reported'`, `expiry_date = today`, `days_until_expiry = 0`) are still populated by this RPC. They are ignored by modern code — `details` is authoritative.

### `driver_tardiness`

**What it means:** A driver has been reported as late for a route session.

**Produced by:** `/api/tardiness` Next.js route (calls `notify_tardiness` RPC in `supabase/migrations/080_add_tardiness_system copy.sql`)  
**recipient_user_id:** NULL (admin-only)  
**Visible to:** Admins (RBAC)  
**Push sent:** No

**`details` shape:**
```json
{
  "tardiness_report_id": 5,
  "driver_id": 12,
  "driver_name": "John Smith",
  "route_id": 3,
  "route_number": "R-07",
  "session_type": "morning",
  "session_date": "2026-04-07",
  "reason": "Traffic on M25",
  "reported_at": "2026-04-07T07:45:00Z"
}
```

> **Note:** Legacy columns (`certificate_type = tardiness_report_id`, `certificate_name = reason`, `expiry_date = session_date`, `days_until_expiry = 0`) are still populated by this RPC. `details` is authoritative.

### `trip_cancellation`

**What it means:** A parent has cancelled a child's trip for a specific date/session. Sent to the parent who made the cancellation.

**Produced by:** `cancel-trip` Edge Function (`supabase/functions/cancel-trip/index.ts`)  
**recipient_user_id:** Parent's `auth.users.id`  
**Visible to:** The cancelling parent + Admins (RBAC)  
**Push sent:** Yes — relay template: "Trip cancelled — [child]'s [session_type] trip has been cancelled."

**`details` shape:**
```json
{
  "passenger_id": 9,
  "passenger_name": "Emily Jones",
  "trip_id": 301,
  "route_id": 3,
  "route_number": "R-07",
  "session_type": "morning",
  "trip_date": "2026-04-07",
  "parent_id": 14,
  "parent_name": "Sarah Jones",
  "reason": "Doctor appointment"
}
```

### `cert_expiry_reminder`

**What it means:** A certificate belonging to a specific employee (driver or PA) or vehicle is expiring soon, is urgent, or has already expired. Sent directly to the certificate owner (driver/PA) or stays admin-only for vehicle certificates.

**Produced by:** `create_cert_expiry_reminders()` RPC — cron job (migration 173)  
**recipient_user_id:** Employee's `auth.users.id` (driver or PA), or NULL for vehicle certificates  
**Visible to:** The employee sees their own; Admins see all via RBAC  
**Push sent:** Yes (only when `recipient_user_id` is set, i.e. driver/PA certs)

**Severity cadence:**  
- `expiring_soon` = 8–30 days out → weekly reminder  
- `urgent` = 1–7 days out → daily reminder  
- `expired` = past expiry, up to 90 days → daily reminder  

**`details` shape:**
```json
{
  "entity_type": "driver",
  "entity_id": 12,
  "cert_type": "driving_licence",
  "cert_name": "Driving Licence",
  "expiry_date": "2026-04-10",
  "severity": "urgent",
  "display_name": "John Smith — Driving Licence",
  "subject_document_id": "uuid-or-omitted"
}
```

`entity_type` values: `vehicle` | `driver` | `assistant` | `employee`

> **Note:** `days_until_expiry` is NOT stored — always derive it at render time from `expiry_date` to avoid stale values.

### `driver_at_stop`

**What it means:** The bus has arrived at a stop where a passenger is expected. Sent to all parents linked to that passenger.

**Produced by:** `trg_notify_parent_on_stop_event` DB trigger on `route_stop_events` INSERT (migration 175)  
**recipient_user_id:** Parent's `auth.users.id`  
**Visible to:** Parents of the passenger  
**Push sent:** Yes — relay template: "Bus arriving — [child]'s bus has arrived at [stop]. Please make your way to the stop."

**`details` shape:**
```json
{
  "route_session_id": 7,
  "route_point_id": 3,
  "passenger_id": 9,
  "passenger_name": "Emily Jones",
  "stop_name": "Oak Street",
  "session_type": "morning",
  "event_type": "arrived",
  "event_at": "2026-04-07T07:55:00Z"
}
```

### `trip_started`

**What it means:** A route session has been started (driver departed). Sent to parents of all passengers assigned to the route.

**Produced by:** `trg_notify_parents_on_session_change` DB trigger on `route_sessions` INSERT/UPDATE (migration 175)  
**recipient_user_id:** Parent's `auth.users.id`  
**Visible to:** Parents of all passengers on the route  
**Push sent:** Yes — relay template: "Trip started — [child]'s [session_type] bus has departed."

**`details` shape:**
```json
{
  "route_session_id": 7,
  "route_id": 3,
  "session_type": "morning",
  "started_at": "2026-04-07T07:30:00Z",
  "passenger_name": "Emily Jones"
}
```

### `trip_completed`

**recipient_user_id:** Parent's `auth.users.id`  
**Visible to:** Parents of the passenger  
**Push sent:** Yes — relay template: "Trip completed — [child]'s trip has completed successfully."

**`details` shape:**

  "route_session_id": 7,

### `child_not_on_trip`

**What it means:** The route session ended but the child was never recorded on the trip (no `picked_up` or `dropped_off` event). Requires parent follow-up.

**Produced by:** `trg_notify_parents_on_session_change` DB trigger (migration 175)  
**recipient_user_id:** Parent's `auth.users.id`  
**Visible to:** Parents  
**Push sent:** Yes — relay template: "Trip ended — child not recorded — [child] was not recorded on today's trip."

**`details` shape:** _(same as `trip_completed`)_

### `new_agreement`

**What it means:** An admin has published a new terms/policy document that the app user must review and accept before they can continue using the app.

**Produced by:** `publish_agreement()` RPC (migration 178)  
**recipient_user_id:** Parent / Driver / PA `auth.users.id`  
**Visible to:** The targeted app user + Admins  
**Push sent:** Yes — relay template: "Action required — There's a new [type] you need to review and agree to."

**`details` shape:**
```json
{
  "agreement_id": "uuid",
  "title": "Updated Terms of Service — Spring 2026",
  "type": "terms_of_service"
}
```

`type` values: `general` | `terms_of_service` | `privacy_policy` | `operational_notice` | `data_protection`

> The full agreement body is NOT stored in `details` — fetch it from the `agreements` table using `agreement_id`.

---

### `admin_broadcast`

**What it means:** An admin has manually sent a push notification to a specific user or a broadcast group via the dashboard "Send Message to App Users" page.

**Produced by:** `send_admin_broadcast()` RPC (migration 179, updated in migration 180)  
**recipient_user_id:** Each targeted user's `auth.users.id` (one row per recipient)  
**Visible to:** The recipient + Admins  
**Push sent:** Yes — relay strips Markdown from `details.body_md` for the FCM lock-screen notification. Firestore stores raw `body_md` so Flutter renders rich text in-app.

**`details` shape (migration 180+):**
```json
{
  "title": "School closure tomorrow",
  "body_md": "Due to **adverse weather**, all morning routes are cancelled on 8 April.\n\nPlease check your app for updates.",
  "audience_type": "route_parents",
  "route_id": 3,
  "sent_by": "uuid-of-admin-user"
}
```

`route_id` is `null` for non-route audience types. `audience_type` is one of:  
`single_user` | `route_parents` | `route_crew` | `all_parents` | `all_drivers` | `all_passenger_assistants`

> **Migration 179 rows** stored `body` (plain text) instead of `body_md`. The relay handles both: it reads `body_md` if present, falls back to `body`.

## Adding a new notification type — checklist

1. Choose a `snake_case` name that clearly describes the event.
2. Insert into `notifications` with `notification_type` set and all payload in `details` JSONB. Do **not** populate the legacy compliance columns.
3. Set `recipient_user_id` to the target user's UUID if this should reach the mobile app. Leave NULL for admin-only.
4. If `recipient_user_id` is non-null, add a `case` to `buildMessage()` in `supabase/functions/push-notification-relay/index.ts` and redeploy.
5. Add an entry to this file.
6. Add the type to the active notification types table in `.github/copilot-instructions.md`.
